A bit of theory: von Neumann Architecture¶

  • The basis for all contemporary computers;
  • RAM is an array of bytes, each having a unique address;
  • A byte is basically a number from 0 to 255 often represented in hexadecmal (00..FF) or binary (00000000..11111111).
  • Program is indistinguishable from data: both are sequences of numbers;
  • CPU interprets sequences of numbers as instructions;
  • The program must be in the RAM to be executed.

Important!¶

  • Programs are just "special data"
  • CPU follows instructions encoded by that "special data"
  • CPU reads programs from RAM ("the memory")
  • To be executed, a program must be in RAM!
  • Disks are not "memory", they are just input/output devices!
  • CPU is fast. Memory is slower. Devices are very slow.

To convert a human-readable (and writeable) program, one needs "translator":¶

Translators can work statement by statement¶

Then it is called "interpreter"¶

  • The program is like a "script" (for the interpreter)
  • The interpreted language is "scripting language"

Iterpreters have advantages and disadvantages.¶

  • Good:
    • No special setup needed: just run the interpreter on your program:
        Rscript myprog.R
    • If program changes, no special actions are needed
    • The program runs on every computer that has the interpreter
  • Bad:
    • Every statement of your program is translated every time:
      • When you run the same program again
      • When you repeat statements in a loop!
    • If there is an error, you will know it when interpreter tries it

Translator can work the whole program at once, keep result¶

This is called "compiler":¶

Corresponding languages are called "compiled languages".

Compilers have advantages and disadvantages.¶

Good:

  • Time saving: translation is done only once;
  • Error catching: mistakes cause compiler to stop;
  • Compiler comprehends your program better and can optimize it (more on this later)

Bad:

  • Time consuming: every change needs recompilation
  • Machine code is quite specific,and will not work:
    • On a different CPU architecture
    • On a different operating system

Middle ground: virtual machines¶

Attempt to get best of both worlds:

  • Compile the whole program every time to a "byte code"
    • A machine code for an imaginary "virtual machine"
  • The byte code may or may not be saved to a file
  • Interpret the byte code
    • "Run" the "virtual machine"

Good:

  • The mistakes are caught at once;
  • Repeated statements run faster;
  • Often-repeated bytecode can be compiled to the real machine code;
    • This is called Just-in-time (JIT) compilation
  • Can run anywhere where virtual machine is installed.

Bad:

  • Virtual machine is not as fast as the real CPU;
  • If the byte code is saved, changes need recompilation;
  • If byte code is not saved, compilation needs to be repeated every time;

Static typing vs. Duck typing¶

Compare these codes:

Python:

def test(a, b):
    c = a+b
    return c

C++:

int test(int a, int b) {
    auto c = a+b;
    return c;
}
  • Python function: a and b can be anything;
    • Can be called with integers, strings, etc.
    • Python has to check every time: what are those a,b?
    • Will work with anything that knows "+" ("duck typing")
  • C++ function: a and b are always integers;
    • Compiled to a few machine code instructions adding two integers;
    • Won't accept strings (you have to write another function);
    • Types have to be known in advance ("static typing")

Optimizing compilers¶

Consider this:

def test(a, b):
    c = a + b
    c = a - b
    c = a + b
    return c

If the translator new that a, b and c are numbers, it could replace the function with a simple a+b. Python has to be careful: if + and - has a special meaning for a and b, all 3 lines are necessary.

Compiled languages are often heavily optimizing the code, greatly speeding it up. Virtual machine based may optimize during JIT compiling. Duck typing is much harder to optimize than static typing!

Excitable medium problem¶

Consider a set of connected nodes, each node has a number of neighbors.

The node can be in 3 states:

  1. Excitable, or "normal" state;
  2. Excited, or "activated" state;
  3. Refractory, or "tired" state.

A normal node become excited if there is enough excited neighbors. After some time, an excited node goes to a refractory state, which cannot be excited. After some rest time, the refractory node becomes normal again.

Example: forest fire. If there are enough burning trees around, a tree will catch fire. Once it's burned down, it cannot catch fire any more until it grows again.

Example: miocard (heart muscle) neural network. Cells are excited by their neighbors, but later become tired and cannot be excited till they rest.

Our implementation is made deliberately general:¶

  • The nodes are arranged in a list (the graph);

  • The nodes are not ordered by proximity: a nearest node is not always next in the list;

  • Node's neighbors can be at different distance (think closest neighbor vs diagonal neighbor).

    17		/// Update the graph in-place, return a tuple with numbers of (unexcited, resting, excited) cells
    18		std::tuple<int,int,int> update(Graph& uv_graph, int delta, uv_type u0, uv_type v0) {
    19		    static const int excitation_fn_coeffs[] = {0, 2, 1}; // indexed by connection "length"
    20		    using intvec=std::vector<int>;
    21		
    22		    int n_calm=0;
    23		    int n_resting=0;
    24		    int n_excited=0;
    25		
    26		    auto ncells=uv_graph.size();
    27		
    28		    // save the u values before update
    29		    intvec old_u(ncells);
    30		    for (auto i=0u; i<ncells; ++i) {
    31		        old_u[i]=uv_graph[i].u;
    32		    }
    33		
    34		    // We do a breadth-first search on a graph
    35		    intvec visited(ncells,false);
    36		    std::deque<int> to_be_processed {0};
    37		
    38		    while (!to_be_processed.empty()) {
    39		        auto idx = to_be_processed.front(); to_be_processed.pop_front();
    40		        if (visited[idx]) continue;
    41		        visited[idx]=true;
    42		
    43		        auto& cell=uv_graph[idx];
    44		        // survey the neighbors, calculate excitation level
    45		        auto excitation=0;
    46		        for (auto cn: cell.conn) {
    47		            to_be_processed.push_back(cn.index);
    48		            excitation += excitation_fn_coeffs[cn.length]*old_u[cn.index];
    49		        }
    50		
    51		        // decide the behavior of the cell:
    52		        auto u = old_u[idx];
    53		        auto v = cell.v;
    54		
    55		        if (u>0) { // cell is still excited?
    56		            --u;
    57		            cell.u = u;
    58		            if (u==0) cell.v = v0; // cell started resting
    59		        }
    60		        else if (v>0) { // cell is resting?
    61		            cell.v = v-1;
    62		        }
    63		        else if (excitation>delta) { // cell becomes excited?
    64		            cell.u = u0;
    65		        }
    66		
    67		        // count the numbers
    68		        if (cell.u==0 && cell.v==0) {
    69		            ++n_calm;
    70		        }
    71		        else if (cell.u>0) {
    72		            ++n_excited;
    73		        } else {
    74		            ++n_resting;
    75		        }
    76		    }

Take-home message:¶

A software application that interacts extensively with I/O devices, for example, reads from and writes to files, or send and receives a lot of data via the network, will experience diminished returns from "fast" languages.
